# AGENTS

## Project

**BiClock (Bilibili 时钟)** — a small Manifest V3 browser extension that overlays the current time on top of the Bilibili video player. By default it shows **while in browser fullscreen and controls are visible**, because only real fullscreen avoids being covered by Bilibili's/browsers' own floating video toolbars; the user can turn off "仅全屏显示" to also show in non-fullscreen player modes. Loads directly as unpacked source; there is no build step, bundler, TypeScript, or test setup in this repo despite the parent directory being `TypeScript/`.

## Layout

- `manifest.json` — MV3 manifest. Content scripts `shared.js` + `biclock.js` (loaded in that order) run on `*://*.bilibili.com/video/*` and `*://*.bilibili.com/bangumi/*`. Declares `action.default_popup` → `popup.html`, and `permissions: ["storage"]`. Includes `browser_specific_settings.gecko.id`, so the same package targets both Firefox and Chromium.
- `shared.js` — the single source of truth for logic shared between the popup and the content script. Provides `DEFAULTS`, `THEME_STYLE_KEYS`, `REMOVED_THEME_IDS`, `pad`, `formatTime`, `makeClockPart`, `hexToRgba`, `renderClockLayout(el, time, style, prefix)`, and `migrateRemovedTheme(config, onPersist)`. Loaded before `popup.js` (popup) and before `biclock.js` (content script) so its top-level `var`/`function` declarations are visible to the dependent script via the shared classic-script global — **no ES modules, no imports**. The `prefix` arg to `renderClockLayout` namespaces child class names: the popup preview and the content script both pass `'bpx-player-clock'` (so user CSS selectors hit identically in both); theme-card samples pass `'clock'` to stay isolated from user CSS.
- `biclock.js` — the content script (loaded after `shared.js`). Creates a `.bpx-player-top-clock` div, formats the time via shared helpers, and appends it to `.bpx-player-container`.
- `popup.html` / `popup.css` / `popup.js` — the extension popup. `popup.html` loads `shared.js` then `popup.js`. Lets the user customize clock styling and time format; writes to `chrome.storage.local` on every change (auto-save), with a live preview driven by a 1-second interval while the popup is open. For the two color fields (文字颜色 / 背景颜色) the popup offers a row of one-click swatches plus a Hex text input (`colorHex` / `bgColorHex`, normalized to `#rrggbb`); each swatch only sets `color` or `backgroundColor`, so the swatch palette is a popup-only convenience over existing keys — it is **not** a new setting key and is intentionally not duplicated into `biclock.js` or `DEFAULTS`. There is no native `<input type="color">` color picker anywhere in the popup. The 主题 grid lists the curated preset themes (in `popup.js` `THEMES`) followed by the user's saved custom themes (from `config.customThemes`); preset cards are immutable `<button>`s, custom cards are `<div role="button">` with two inline action buttons — ↻ 更新 (overwrite that theme's appearance keys with the current config) and × 删除 — and a name span that enters an inline rename input on double-click. The 外观 fieldset's bottom button saves the current appearance as a new custom theme (`id: "custom_<timestamp>"`, name defaulted to "自定义 N" via `nextCustomName()`); there is **no** "重置为默认" button — clicking the Bilibili 粉 preset card achieves the same thing. Only the `THEME_STYLE_KEYS` (the 12 appearance fields) flow into/out of custom themes; position and display switches (`posX`/`posY`, `fullscreenOnly`, `alwaysShow`) are never part of a theme. A 自定义 CSS fieldset offers an 启用 toggle + `<textarea id="customCss">` for arbitrary CSS overlaid on top of the active theme; the legend's `?` bubble lists the available `.bpx-player-top-clock` / `.bpx-player-clock-*` selectors and reminds the user that overriding inline styles needs `!important`. The textarea is bound separately from `onInput` because editing CSS must **not** flip `clockStyle` to `'custom'` (CSS is decoupled from themes); the toggle goes through the normal `onInput` path. The preview clock node carries `bpx-player-top-clock` and a `<style id="preview-custom-css">` in `document.head` mirrors the on-page injection so the preview reflects user CSS.
- `icons/clock.png` — extension icon (size 200).

## Commands

There is no package manager and no `package.json` deps (the file exists only for repo metadata). Two plain-Node helper scripts live at the repo root:

- `node package.mjs` — zips the runtime files (`manifest.json`, `shared.js`, `biclock.js`, `popup.html`, `popup.css`, `popup.js`, `icons/clock.png`) into `BiClock-<manifest-version>.zip`, with files at the archive root (no nested folder) for store upload. Cross-platform: prefers system `zip`, falls back to `tar` (Windows ships bsdtar as `tar.exe`); `--force-local` is passed to `tar` so Windows drive letters like `D:` aren't parsed as rsh-style hosts.
- `.github/workflows/release.yml` — on pushing a `v*` tag, verifies the tag matches `manifest.json` `version`, runs `node package.mjs`, and creates a GitHub Release with `BiClock-<version>.zip` attached; the Release body is auto-generated from commits between the previous tag and this one. Tag must equal the manifest version or CI fails fast.

To test changes locally:

1. Load the folder as an unpacked/temporary extension (chrome://extensions in dev mode, or `about:debugging` → This Firefox → Load Temporary Add-on).
2. Open any Bilibili `/video/` or `/bangumi/` page, enter the player, **enter browser fullscreen**, and move the mouse to show controls — with defaults (仅全屏显示 on, 常驻显示 off) the clock appears centered at the top only while both `data-screen="full"` and `data-ctrl-hidden="false"`.

There is nothing to typecheck or lint; review changes by reading the file. `node --check <file.js>` catches syntax errors for the three JS files.

## Architecture & coupling notes

- `biclock.js` is tightly coupled to Bilibili's current player DOM. It queries specific class names that can change when Bilibili ships a player update:
  - `.bpx-player-container` — the element observed for control visibility and screen mode.
  - `.bpx-player-top-left` — the sibling the clock is inserted after.
  - Attribute `data-screen` on `.bpx-player-container` — `"full"` means true browser fullscreen (Fullscreen API). Only this mode hides Bilibili's/browsers' own floating video toolbars, so by default (仅全屏显示 on) the clock is shown **only** when `data-screen="full"`. When the user turns 仅全屏显示 off, this attribute is ignored and the clock shows in any screen mode (normal, wide, web, full) — the user accepts coexisting with those overlays. Wide (`"wide"`) and web fullscreen (`"web"`) modes are otherwise intentionally excluded.
  - Attribute `data-ctrl-hidden` on `.bpx-player-container` — `"false"` means controls visible. In the default **mouse-triggered** mode (`alwaysShow=false`), this must be `"false"` for the clock to show. In **always-on** mode (`alwaysShow=true`) it is ignored. `shouldShow()` is the single gatekeeper; it first applies the `fullscreenOnly` gate, then the `alwaysShow` gate.
  - The older `bilibili-player` (`.bilibili-player-`) class prefix is **not** handled; only the current `bpx-player-` prefix is. If the clock stops appearing, the first thing to check is whether Bilibili renamed these classes.
- A `MutationObserver` watches attribute changes on `.bpx-player-container`; the 1-second `setInterval` is started only when `shouldShow()` is true and stopped otherwise. `shouldShow()` requires `data-screen="full"` only when `fullscreenOnly` is on; it additionally requires `data-ctrl-hidden="false"` unless `alwaysShow` is on. `stopTimer()` also detaches the clock node from the DOM so it doesn't linger when it shouldn't show. Don't introduce a second always-running interval — keep it tied to `shouldShow()`. Display-setting switches (`fullscreenOnly` / `alwaysShow`) arrive via `chrome.storage.onChanged` (not a player attribute), so the change handler re-runs `shouldShow()` and starts/stops the timer immediately.
- `updateClock()` re-inserts the clock node on every tick. The clock node is reused (same reference), so this moves rather than duplicates it; if you change this, preserve the "single reused node" invariant to avoid stacking duplicates. It is always appended to `.bpx-player-container` (both display modes). The node is positioned with `position: fixed` relative to the **player container's viewport rect** (`getBoundingClientRect`), not relative to the viewport as a whole — the player does not fill the viewport in non-fullscreen modes (Bilibili chrome takes up the top/sides), so a viewport-relative `posX/posY` ratio would place the clock outside the video. Container-relative coordinates keep the clock inside the video across normal/wide/web/fullscreen modes. Mounting on the container (rather than inside the top bar) also avoids the CSS transform/contain trap where a `fixed` descendant is re-anchored to a transformed ancestor; the container is a stable host. `.bpx-player-top-left` is no longer used as a mount point.
- Styling is applied imperatively via `clock.style.*`. There is no CSS file injected into the page and no shadow DOM. (The popup has its own `popup.css`, but the on-page clock is still styled imperatively.) **Exception**: user-supplied `customCss` is injected via a single `<style id="biclock-custom-css">` element appended to `document.documentElement` (not `head` — `head` may not exist yet at content-script run time). The element is created once; `applyCustomCss()` rewrites its `textContent` on every `applyStyles()` call (so it rides the existing 1-second tick + `chrome.storage.onChanged` lifecycle — no new timer or observer). The CSS applies document-wide but only `.bpx-player-top-clock` / `.bpx-player-clock-*` selectors match anything; because the built-in styles are inline, overriding them requires `!important`. `customCssEnabled=false` or empty string clears the `textContent` (the node stays). The popup mirrors this with its own `<style id="preview-custom-css">` so the preview reflects user CSS; to make selectors hit, the preview clock node carries `bpx-player-top-clock` and `refreshPreviewText()` passes prefix `'bpx-player-clock'` (theme-card samples still use `'clock'` to stay isolated).
- User styling flows through `chrome.storage.local`. `popup.js` writes settings on every input event (auto-save, no Save button). `biclock.js` reads them at startup and subscribes to `chrome.storage.onChanged` so changes take effect on an already-open Bilibili tab without a reload.
- `DEFAULTS` lives in `shared.js` as the single source of truth for default values — it is no longer duplicated, so changing a default or adding a field is a one-file edit. The keys are: `fontSize`, `color`, `backgroundColor`, `bgOpacity` (0–100), `bold`, `fullscreenOnly` (scope gate: true = browser-fullscreen only, the default; false = show in any player screen mode), `alwaysShow` (display mode within that scope: false = mouse-triggered, true = always-on), `posX`/`posY` (clock position as a 0–1 ratio of the **player container's** width/height, **edge-aligned**: `(posX, posY)` is where the clock's top-left corner lands — `0/0` puts it at the container's top-left, `1/1` puts its bottom-right at the container's bottom-right, `0.5/0.5` is centered; the popup's "位置" panel is the sole source of position — there is no separate top-offset knob), `customThemes` (an array of user-saved custom themes; default `[]`; **only the popup reads or writes it** — the content script never consumes it, so adding/removing custom themes has zero effect on the running clock unless one becomes the active `clockStyle`), and `customCss` / `customCssEnabled` (user-supplied arbitrary CSS; default `''` / `false`; **not** part of `THEME_STYLE_KEYS` — decoupled from themes, so switching/deleting themes never touches it, and theme cards never carry it). Each custom theme element is `{ id, name }` plus the 12 `THEME_STYLE_KEYS`; `id` is `"custom_<timestamp>"` and is also stored in `clockStyle` when that theme is active, `name` defaults to `"自定义 N"` (auto-numbered to avoid colliding with already-renamed cards) and is editable via double-click on the card name. Background color is stored as `#rrggbb` plus a separate `bgOpacity` percentage; the default `backgroundColor` is `#fb7299` (Bilibili pink), with `color` defaulting to `#ffffff`. Both popup and content script convert to `rgba()` via the shared `hexToRgba()` helper. The time format is hardcoded to 24-hour with seconds (`HH:MM:SS`) and is no longer configurable, so neither `use24Hour` nor `showSeconds` exists as a setting.

## Conventions

- Vanilla JS, `var`/`function` style, browser globals only — no modules, no imports. Match this style for the content scripts (`shared.js`, `biclock.js`); they run as classic scripts whose top-level `var`/`function` declarations share one global, so `biclock.js` may reference symbols defined in `shared.js` purely by load order (manifest's `content_scripts.js` array is `["shared.js", "biclock.js"]`).
- Keep all DOM queries defensive: Bilibili pages are SPA-driven, so elements may not exist yet. `run()` currently assumes `.bpx-player-container` is present at `DOMContentLoaded` / on script run; if you add features, re-query at use time rather than caching long-lived references.
