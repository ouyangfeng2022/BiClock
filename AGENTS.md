# AGENTS

## Project

**BiClock (Bilibili 时钟)** — a small Manifest V3 browser extension that overlays the current time on top of the Bilibili video player. By default it shows **while in browser fullscreen and controls are visible**, because only real fullscreen avoids being covered by Bilibili's/browsers' own floating video toolbars; the user can turn off "仅全屏显示" to also show in non-fullscreen player modes. Loads directly as unpacked source; there is no build step, bundler, TypeScript, or test setup in this repo despite the parent directory being `TypeScript/`.

## Layout

- `manifest.json` — MV3 manifest. Content scripts `shared.js` + `biclock.js` (loaded in that order) run on `*://*.bilibili.com/video/*` and `*://*.bilibili.com/bangumi/*`. Declares `action.default_popup` → `popup.html`, `options_ui` → `options.html` (with `open_in_tab: true`, so the settings page opens as a full browser tab rather than a second popup), and `permissions: ["storage"]`. Includes `browser_specific_settings.gecko.id`, so the same package targets both Firefox and Chromium.
- `shared.js` — the single source of truth for logic shared between the popup, the options page, and the content script. Provides `DEFAULTS`, `THEME_STYLE_KEYS`, `THEME_CSS_KEYS`, `APPEARANCE_INLINE_KEYS`, `REMOVED_THEME_IDS`, `pad`, `formatTime`, `makeClockPart`, `hexToRgba`, `renderClockLayout(el, time, style, prefix)`, and `migrateRemovedTheme(config, onPersist)`. Loaded before `popup.js` (popup), before `options.js` (options page), and before `biclock.js` (content script) so its top-level `var`/`function` declarations are visible to the dependent script via the shared classic-script global — **no ES modules, no imports**. `THEME_STYLE_KEYS` (12 appearance keys) and `THEME_CSS_KEYS` (`customCss` / `customCssEnabled`) together define what a custom theme snapshots; only `options.js` consumes `THEME_CSS_KEYS`. `APPEARANCE_INLINE_KEYS` is the shared list of inline-style properties that `applyStyles()` / `applyToPreview()` wipe when CSS mode is on (so user CSS is the sole appearance source — see the Styling bullet below). The `prefix` arg to `renderClockLayout` namespaces child class names: the popup preview, the options preview, and the content script all pass `'bpx-player-clock'` (so user CSS selectors hit identically in all three); theme-card samples pass `'clock'` to stay isolated from user CSS.
- `biclock.js` — the content script (loaded after `shared.js`). Creates a `.bpx-player-top-clock` div, formats the time via shared helpers, and appends it to `.bpx-player-container`.
- `popup.html` / `popup.css` / `popup.js` — the extension popup, now a **「快速调整」subset**. `popup.html` loads `shared.js` then `popup.js`. The popup shows only: the live preview (sticky), an 外观 fieldset (字体大小 / 粗体 / 文字颜色 + swatches / 背景颜色 + swatches / 透明度), and a 显示 fieldset (仅全屏显示 / 常驻显示 + the `?` help bubble with active-state highlighting). A header button 「⚙ 更多设置」 and a bottom hint link both call `chrome.runtime.openOptionsPage()` to open the full settings page in a new tab. Writes to `chrome.storage.local` on every change (auto-save). For the two color fields the popup offers a row of one-click swatches plus a Hex text input (`colorHex` / `bgColorHex`, normalized to `#rrggbb`); each swatch only sets `color` or `backgroundColor`, so the swatch palette is a popup-only convenience over existing keys — it is **not** a new setting key and is intentionally not duplicated into `biclock.js` or `DEFAULTS`. There is no native `<input type="color">` color picker anywhere. The popup **no longer** hosts the 主题 grid, the 位置 panel, the 自定义 CSS textarea, or the 保存为自定义主题 button — all of those moved to the options page. It still reads `customCss` from storage to drive the preview (so the preview matches the real clock), but it does **not** write `customCss` / `customThemes` / `posX` / `posY` / `clockStyle` unless the user actually edits a popup field, so it won't clobber a concurrent edit on the options page. The preview clock node carries `bpx-player-top-clock` and a `<style id="preview-custom-css">` in `document.head` mirrors the on-page injection so the preview reflects user CSS.
- `options.html` / `options.css` / `options.js` — the full **设置页 (options page)**, opened as a browser tab via `options_ui` (reached from the popup's 「更多设置」 button or the browser's extension-options entry — right-click icon → 选项 in Chromium, about:addons → BiClock → 设置 in Firefox). `options.html` loads `shared.js` then `options.js`. The page is a **three-column layout** (body `max-width: 1240px`): a left **分区导航** (`.options-nav`, sticky at `top:16px`, ~200px wide, AdGuard-style: no outer card border — sits flush on the page bg; each `.nav-item` is an icon glyph in a `.nav-icon` span + label, hover gets a `--surface-2` wash, and the active item gets a `--pink-soft` background, pink icon, bolder text, and a 3px pink bar — a left vertical bar in the side-column mode, switching to a bottom underline when the nav collapses to a horizontal tab strip at narrower widths — four anchor links `#sec-theme` / `#sec-appearance` / `#sec-css` / `#sec-about` pointing at the 主题 / 外观+显示 `.card-group` / 自定义 CSS / 关于 fieldsets; an `IntersectionObserver` scroll-spy in `options.js` `initNavSpy()` adds `.is-active` to whichever section is currently in the viewport's active band — older browsers without `IntersectionObserver` silently fall back to highlighting only the first item, anchor jumps still work via `<a href="#...">` + `scroll-behavior: smooth`), a middle scrolling **控制面板** (`.options-main`), and a right **预览栏** (`.preview-column`). The preview is a 16:9 `.preview-banner` whose background is a **pure-CSS reconstruction of the Bilibili player shell** (top bar with 圆形头像占位 / UP主名 / 粉色关注 button, a faint grid "video" region, and a bottom control bar with CSS-drawn play triangle / timecode / 粉色 progress fill / volume / fullscreen glyphs — all `aria-hidden="true"` decoration, no real assets), so the clock visually sits on a representative Bilibili player. The preview clock (`#previewClock`, `.bpx-player-top-clock`) is `position: absolute` inside the banner and **directly draggable** — clicking anywhere on the banner or dragging the clock sets `posX/posY` (the separate 位置 fieldset that used to exist has been removed; position control is now fused into the preview). Position math mirrors `biclock.js` exactly: `left/top` in % plus `transform: translate(posX*-100%, posY*-100%)` for edge-aligned semantics (posX=0 left-edge, 1 right-edge, 0.5 centered). The banner also carries a `实时预览` pill (top-left), a `拖动时钟调整位置` hint, and a 居中 reset button (`#resetPosition`, bottom-right). The middle column holds: the 主题 grid (the curated preset themes in `options.js` `THEMES` followed by the user's saved custom themes from `config.customThemes`; preset cards are immutable `<button>`s, custom cards are `<div role="button">` with ↻ 更新 / × 删除 inline action buttons and a double-click-to-rename name span) with a `theme-hint` line; a `.card-group` (grid `1fr / 1fr`) holding the 外观 fieldset (same fields as popup: 字体大小 / 粗体 / 文字颜色 + swatches / 背景颜色 + swatches / 透明度) and the 显示 fieldset (same as popup, with the `?` help bubble); the 自定义 CSS fieldset (启用 toggle + `<textarea id="customCss">` + an inline **docs section**: overview paragraph explaining the two-mode model (CSS on ⇒ no inline appearance ⇒ no `!important` needed; position still drag-controlled, layout DOM still JS-rendered), the full `.bpx-player-top-clock` / `.bpx-player-clock-*` selector reference as a two-column table with per-selector descriptions, and 3–4 copy-to-clipboard example cards — 粉色圆角胶囊 / 霓虹光晕终端风 / 脉冲呼吸动画 / 描边数字透明背景, each a complete-look snippet with **no `!important`**); and the 关于 fieldset (project blurb + single `about-meta` line with version / MIT / inline text links). A bottom `.bottom-actions` row holds the single primary button 「保存当前外观与 CSS 为自定义主题」 (`#saveCustomTheme`, `.primary-btn`) — it was moved out of the 主题 fieldset because it now saves appearance **and** CSS, so it belongs to the whole page, not the theme module. At `max-width: 1100px` the three columns collapse to two: the left nav stops being a sticky side column and becomes a horizontal tab strip (`flex-direction: row`, `grid-column: 1 / -1`) pinned above the control panel, while the preview column stays on the right; at `max-width: 920px` it further collapses to one column (nav tab strip on top, then preview, then control panel — preview no longer sticky); at `max-width: 620px` the `.card-group` also collapses. **Custom-theme semantics now include CSS**: `saveCurrentAsCustomTheme()` / `updateCustomTheme()` snapshot the 12 `THEME_STYLE_KEYS` **plus** `THEME_CSS_KEYS` (`customCss` / `customCssEnabled`); `applyTheme()` restores them for custom themes, and for **preset** themes explicitly clears `customCss=''` / `customCssEnabled=false` (presets ship no CSS). Editing the CSS textarea or toggling 启用 flips `clockStyle='custom'` (CSS edits are now theme-relevant). `options.css` is **self-contained** (own `:root` tokens copy, does **not** import `popup.css`). Copy buttons use `navigator.clipboard.writeText` with a 「✓ 已复制」 flash and a fallback to selecting the code text. `options.js` is effectively the pre-split `popup.js` plus the copy-button logic and the left-nav scroll-spy (`initNavSpy()`); the swatch / Hex / preview / form helpers are deliberately duplicated between `popup.js` and `options.js` rather than extracted into a shared module, because the two pages have subtly different field sets and side effects (the popup, for instance, does not read or write `customCss` / `customCssEnabled` at all — it only mirrors `customCss` for its preview).
- `icons/clock.png` — extension icon (size 200).
- `shot.mjs` + `popup-shot.png` / `options-shot.png` — Playwright script that regenerates the two README screenshots and its output PNGs. **Dev-only artifacts** (not part of the runtime package — `package.mjs` INCLUDE list intentionally excludes them, same as `README.md` / `AGENTS.md` / `node_modules`). The PNGs are committed so the README renders on GitHub without a build step. See Commands below for how to run it and Conventions for the "re-shoot after UI change" rule.

## Commands

There is no package manager and no `package.json` deps (the file exists only for repo metadata). Two plain-Node helper scripts live at the repo root:

- `node package.mjs` — zips the runtime files (`manifest.json`, `shared.js`, `biclock.js`, `popup.html`, `popup.css`, `popup.js`, `options.html`, `options.css`, `options.js`, `icons/clock.png`) into `BiClock-<manifest-version>.zip`, with files at the archive root (no nested folder) for store upload. Cross-platform: prefers system `zip`, falls back to `tar` (Windows ships bsdtar as `tar.exe`); `--force-local` is passed to `tar` so Windows drive letters like `D:` aren't parsed as rsh-style hosts. Missing any file in the INCLUDE list fails the script fast.
- `.github/workflows/release.yml` — on pushing a `v*` tag, verifies the tag matches `manifest.json` `version`, runs `node package.mjs`, and creates a GitHub Release with `BiClock-<version>.zip` attached; the Release body is auto-generated from commits between the previous tag and this one. Tag must equal the manifest version or CI fails fast.
- `node shot.mjs [popup|options|all]` — regenerates the README screenshots `popup-shot.png` and `options-shot.png` (default `all` does both) by loading `popup.html` / `options.html` in Playwright Chromium and capturing a full-page screenshot. **The screenshots must stay in sync with the actual UI.** The script injects a pure-in-memory `chrome.storage.local` / `chrome.runtime` mock via `page.addInitScript` *before* each page loads — without it, `popup.js` / `options.js` `init()` throws on `chrome.storage.local.get(...)` under `file://` (the `chrome` global doesn't exist outside an extension), so no form is filled, no swatches are built, and the preview clock never gets its inline styles, which is exactly why the old popup screenshot looked nothing like the real popup. The mock makes `init()` run end-to-end, so the screenshot equals the real rendered page. Output paths are derived from the script location (cross-platform, no hardcoded drive letter). Depends on the dev-only `playwright` in `node_modules`; if the Chromium binary is missing, run `npx playwright install chromium` first.

To test changes locally:

1. Load the folder as an unpacked/temporary extension (chrome://extensions in dev mode, or `about:debugging` → This Firefox → Load Temporary Add-on).
2. Open any Bilibili `/video/` or `/bangumi/` page, enter the player, **enter browser fullscreen**, and move the mouse to show controls — with defaults (仅全屏显示 on, 常驻显示 off) the clock appears centered at the top only while both `data-screen="full"` and `data-ctrl-hidden="false"`.

There is nothing to typecheck or lint; review changes by reading the file. `node --check <file.js>` catches syntax errors for the four JS files (`shared.js`, `biclock.js`, `popup.js`, `options.js`).

## Architecture & coupling notes

- `biclock.js` is tightly coupled to Bilibili's current player DOM. It queries specific class names that can change when Bilibili ships a player update:
  - `.bpx-player-container` — the element observed for control visibility and screen mode.
  - `.bpx-player-top-left` — the sibling the clock is inserted after.
  - Attribute `data-screen` on `.bpx-player-container` — `"full"` means true browser fullscreen (Fullscreen API). Only this mode hides Bilibili's/browsers' own floating video toolbars, so by default (仅全屏显示 on) the clock is shown **only** when `data-screen="full"`. When the user turns 仅全屏显示 off, this attribute is ignored and the clock shows in any screen mode (normal, wide, web, full) — the user accepts coexisting with those overlays. Wide (`"wide"`) and web fullscreen (`"web"`) modes are otherwise intentionally excluded.
  - Attribute `data-ctrl-hidden` on `.bpx-player-container` — `"false"` means controls visible. In the default **mouse-triggered** mode (`alwaysShow=false`), this must be `"false"` for the clock to show. In **always-on** mode (`alwaysShow=true`) it is ignored. `shouldShow()` is the single gatekeeper; it first applies the `fullscreenOnly` gate, then the `alwaysShow` gate.
  - The older `bilibili-player` (`.bilibili-player-`) class prefix is **not** handled; only the current `bpx-player-` prefix is. If the clock stops appearing, the first thing to check is whether Bilibili renamed these classes.
- A `MutationObserver` watches attribute changes on `.bpx-player-container`; the 1-second `setInterval` is started only when `shouldShow()` is true and stopped otherwise. `shouldShow()` requires `data-screen="full"` only when `fullscreenOnly` is on; it additionally requires `data-ctrl-hidden="false"` unless `alwaysShow` is on. `stopTimer()` also detaches the clock node from the DOM so it doesn't linger when it shouldn't show. Don't introduce a second always-running interval — keep it tied to `shouldShow()`. Display-setting switches (`fullscreenOnly` / `alwaysShow`) arrive via `chrome.storage.onChanged` (not a player attribute), so the change handler re-runs `shouldShow()` and starts/stops the timer immediately.
- `updateClock()` re-inserts the clock node on every tick. The clock node is reused (same reference), so this moves rather than duplicates it; if you change this, preserve the "single reused node" invariant to avoid stacking duplicates. It is always appended to `.bpx-player-container` (both display modes). The node is positioned with `position: fixed` relative to the **player container's viewport rect** (`getBoundingClientRect`), not relative to the viewport as a whole — the player does not fill the viewport in non-fullscreen modes (Bilibili chrome takes up the top/sides), so a viewport-relative `posX/posY` ratio would place the clock outside the video. Container-relative coordinates keep the clock inside the video across normal/wide/web/fullscreen modes. Mounting on the container (rather than inside the top bar) also avoids the CSS transform/contain trap where a `fixed` descendant is re-anchored to a transformed ancestor; the container is a stable host. `.bpx-player-top-left` is no longer used as a mount point.
- Styling is applied via a **two-mode** path controlled by `customCssEnabled && customCss` (the "cssMode" gate). **Inline mode** (default, CSS off): appearance is applied imperatively via `clock.style.*` on every tick — no injected CSS file, no shadow DOM. **CSS mode** (CSS on): `applyStyles()` instead walks `APPEARANCE_INLINE_KEYS` (a shared.js constant — `color, backgroundColor, fontWeight, fontFamily, fontSize, textShadow, border, padding, borderRadius, boxSizing`) and calls `removeProperty` on each, so user CSS becomes the **sole** source of appearance and **no `!important` is needed** to override the built-in look. Position (`position/left/top/transform/zIndex/userSelect`) is always JS-computed in both modes (so the options-page drag panel keeps working in CSS mode); complex-layout DOM (analog dial hands, flip cards, etc.) is always JS-rendered in both modes (those child coordinates can't be rebuilt in CSS). Because `updateClock()` re-runs `applyStyles()` every tick, switching the 启用 toggle or editing the CSS textarea re-cleans/re-applies inline each second — no flicker, no stale inline leak. (The popup has its own `popup.css` and the options page has its own `options.css`, but the on-page clock follows this two-mode rule.) User-supplied `customCss` is injected via a single `<style id="biclock-custom-css">` element appended to `document.documentElement` (not `head` — `head` may not exist yet at content-script run time). The element is created once; `applyCustomCss()` rewrites its `textContent` on every `applyStyles()` call (so it rides the existing 1-second tick + `chrome.storage.onChanged` lifecycle — no new timer or observer). The CSS applies document-wide but only `.bpx-player-top-clock` / `.bpx-player-clock-*` selectors match anything. `customCssEnabled=false` or empty string clears the `textContent` (the node stays) and falls back to inline mode. Both the popup and the options page mirror this with their own `<style id="preview-custom-css">` and the same two-mode `applyToPreview()`, so each preview reflects user CSS identically; to make selectors hit, both preview clock nodes carry `bpx-player-top-clock` and both `refreshPreviewText()` calls pass prefix `'bpx-player-clock'` (theme-card samples still use `'clock'` to stay isolated). The options page is the only place that *edits* `customCss`; the popup still mirrors it so its preview stays consistent with the real clock.
- User styling flows through `chrome.storage.local`. `popup.js` and `options.js` both write settings on every input event (auto-save, no Save button); `biclock.js` reads them at startup and subscribes to `chrome.storage.onChanged` so changes from **either** UI take effect on an already-open Bilibili tab without a reload. The popup and the options page never message each other directly — they coordinate purely through storage (the popup re-reads storage every time it opens, and `chrome.storage.onChanged` is the only cross-page channel the content script needs). Because both UIs can be open at once, avoid writing keys you didn't read from a control: `popup.js` deliberately calls `save()` only after a real field edit, so it won't overwrite `customThemes` / `customCss` / `posX` / `posY` changes made concurrently on the options page. The popup's 「更多设置」 button and bottom hint link both call `chrome.runtime.openOptionsPage()` (the standard MV3 entry; do **not** use a `web_accessible_resources` + manual-tab approach — `options_ui` registration also gives you the browser's built-in options entry).
- `DEFAULTS` lives in `shared.js` as the single source of truth for default values — it is no longer duplicated, so changing a default or adding a field is a one-file edit. The keys are: `fontSize`, `color`, `backgroundColor`, `bgOpacity` (0–100), `bold`, `fullscreenOnly` (scope gate: true = browser-fullscreen only, the default; false = show in any player screen mode), `alwaysShow` (display mode within that scope: false = mouse-triggered, true = always-on), `posX`/`posY` (clock position as a 0–1 ratio of the **player container's** width/height, **edge-aligned**: `(posX, posY)` is where the clock's top-left corner lands — `0/0` puts it at the container's top-left, `1/1` puts its bottom-right at the container's bottom-right, `0.5/0.5` is centered; position is set by dragging the clock directly on the options-page preview, or by the legacy popup position panel if present — there is no separate top-offset knob), `customThemes` (an array of user-saved custom themes; default `[]`; **only the options page reads or writes it** — the content script never consumes it, so adding/removing custom themes has zero effect on the running clock unless one becomes the active `clockStyle`), and `customCss` / `customCssEnabled` (user-supplied arbitrary CSS; default `''` / `false`). Each custom theme element is `{ id, name }` plus the 12 `THEME_STYLE_KEYS` **and** the 2 `THEME_CSS_KEYS` (`customCss` / `customCssEnabled`) — saving a custom theme snapshots the current CSS alongside the appearance, and applying that theme restores both; preset themes carry no CSS and applying one clears the current CSS. `id` is `"custom_<timestamp>"` and is also stored in `clockStyle` when that theme is active, `name` defaults to `"自定义 N"` (auto-numbered to avoid colliding with already-renamed cards) and is editable via double-click on the card name. Background color is stored as `#rrggbb` plus a separate `bgOpacity` percentage; the default `backgroundColor` is `#fb7299` (Bilibili pink), with `color` defaulting to `#ffffff`. Both popup and content script convert to `rgba()` via the shared `hexToRgba()` helper. The time format is hardcoded to 24-hour with seconds (`HH:MM:SS`) and is no longer configurable, so neither `use24Hour` nor `showSeconds` exists as a setting.

## Conventions

- Vanilla JS, `var`/`function` style, browser globals only — no modules, no imports. Match this style for the content scripts (`shared.js`, `biclock.js`); they run as classic scripts whose top-level `var`/`function` declarations share one global, so `biclock.js` may reference symbols defined in `shared.js` purely by load order (manifest's `content_scripts.js` array is `["shared.js", "biclock.js"]`).
- Keep all DOM queries defensive: Bilibili pages are SPA-driven, so elements may not exist yet. `run()` currently assumes `.bpx-player-container` is present at `DOMContentLoaded` / on script run; if you add features, re-query at use time rather than caching long-lived references.
- **Keep the README screenshots in sync with the UI.** Any change to `popup.html` / `popup.css` / `popup.js` or `options.html` / `options.css` / `options.js` (and any `shared.js` change that affects the rendered preview — `DEFAULTS`, `THEMES`, `renderClockLayout`, the swatch palettes) must be followed by re-running `node shot.mjs` (regenerates both `popup-shot.png` and `options-shot.png`) and committing the updated PNGs in the same commit as the source change. The screenshots are how the README sells the extension, so a stale shot that drifts from the real UI is a bug. Do not hand-edit the PNGs; the script is the single source. If you change the popup/options viewport contract (e.g. body width, three-column breakpoints), also revisit `shot.mjs`'s hardcoded viewport sizes (`320×800` popup, `1280×900` options) so the shot still frames the page the way a real user sees it.
